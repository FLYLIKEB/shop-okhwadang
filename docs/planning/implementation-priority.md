# 구현 우선순위

상위 문서: [[README|옥화당 기획서 홈]]
관련 문서: [[service-plan|서비스 기획서]], [[feature-policy-map|기능별 정책 지도]]

이 문서는 기존 로드맵과 아키텍처 문서를 내부 실행 기준으로 재정리한 것이다.

## 단계별 목표

| 단계 | 목표 | 핵심 산출물 |
|------|------|-------------|
| Phase 1 MVP | 쇼핑몰 핵심 탐색/구매 진입 | 메인, 상품 목록/상세, 장바구니, 기본 인증 |
| Phase 2 Core | 주문/결제/회원 플로우 완성 | 주문 생성, Mock PG, 주문 완료, 마이페이지 |
| Phase 3 Payment/Shipping | 실제 결제와 배송 운영 기반 | 국내 네이버페이/KG이니시스/토스페이먼츠, 글로벌 Stripe, 배송사 어댑터, 운송장/배송 조회 |
| Phase 4 Admin | 운영자 업무 처리 | 상품/주문/배송/회원/쿠폰 관리 |
| Phase 5 CMS | 운영자 편집성 강화 | 블록 기반 페이지, 네비게이션 DB 관리 |
| Phase 6 Growth | 구매 전환/확장 기능 | 리뷰, 위시리스트, 쿠폰, 한국어/영어 글로벌, 외부 연동 |
| Phase 7 Ops | 운영 안정화 | CI/CD, 모니터링, E2E, 성능/보안 점검 |

## MVP 기준

MVP는 실제 고객이 상품을 탐색하고 구매 의사를 완료할 수 있는 최소 범위다.

필수 포함:

- 메인 페이지
- 상품 목록과 상세
- 장바구니
- 회원가입/로그인
- 주문 생성
- Mock 결제
- 주문 완료 화면
- 기본 관리자 상품/주문 확인

MVP에서 미룰 수 있는 항목:

- 실제 PG 전체 연동 중 네이버페이/KG이니시스 어댑터
- 스마트스토어 리뷰 동기화는 별도 이슈: https://github.com/FLYLIKEB/shop-okhwadang/issues/720
- 인스타그램 쇼핑
- 고급 CMS 블록 편집
- 한국어/영어 및 KRW/USD 전체 운영
- 알림톡/SMS 자동화

## 구현 순서 원칙

- 구매 플로우를 먼저 연결하고, 운영 편의 기능은 그 다음에 붙인다.
- PG/배송/스토리지는 어댑터 경계부터 고정하고 실제 외부 연동은 뒤에 붙인다.
- 할인, 포인트, 배송비처럼 금액에 영향을 주는 정책은 결제 전에 서버 기준으로 확정한다.
- 어드민 기능은 프론트 관리 화면보다 백엔드 권한/상태 전이 정책을 먼저 확정한다.
- CMS는 페이지 렌더링 계약과 블록 JSON 스키마를 먼저 정한 뒤 편집 UI를 만든다.

## 리스크

| 리스크 | 대응 기준 |
|--------|-----------|
| PG 심사 지연 | Mock PG로 구매 플로우를 먼저 완성하고 실제 PG는 어댑터로 교체 |
| 국내 PG 범위 확장 | 네이버페이/KG이니시스/토스페이먼츠를 모두 목표 지원하되 어댑터별 심사/검증을 분리 |
| Stripe enum 불일치 | `PaymentGatewayType.STRIPE` 추가와 DB 마이그레이션 필요 |
| 배송 정책 미확정 | 배송비, 해외 배송, 상태 동기화 규칙을 주문 정책보다 먼저 확정 |
| CMS 범위 확장 | MVP에서는 고정 블록 렌더링부터 시작하고 편집 UX는 후순위 |
| 글로벌 스토어 범위 확장 | 데이터 모델과 표시 포맷을 먼저 정하고 번역/환율 자동화는 후순위 |
| 보안 누락 | 인증, RBAC, ValidationPipe, 결제 금액 검증은 MVP에서도 제외하지 않음 |

## 다음 확정 필요 항목

- 네이버페이/KG이니시스 어댑터 구현 일정.
- 해외 배송 가능 국가와 배송비 테이블.
- 주문 취소/환불 시 재고 복구 자동화 구현 범위.
- 감사 로그 자동 삭제/아카이빙 배치 구현 일정.
- CMS 블록 JSON 스키마와 발행/미리보기 정책.
